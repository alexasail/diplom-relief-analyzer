# ReliefAnalyzer

Telegram-бот для анализа рельефа по DEM с двумя режимами результата:
- карта высот и уклонов (PNG);
- 3D-модель рельефа (STL 10x10 см) с платформой-основанием для печати.

Основной источник данных: [OpenTopography Global DEM API](https://portal.opentopography.org/apidocs/).

---

## 1) Что умеет приложение

- Принимает **область по двум углам**: левый верхний и правый нижний.
- Автоматически загружает DEM из OpenTopography с fallback по источникам (`SRTMGL1 -> COP30 -> ASTER`).
- Вычисляет карту уклонов (метод Хорна, 3x3).
- Генерирует PNG с двумя панелями:
  - карта высот (`terrain`);
  - карта уклонов (`coolwarm`).
- Генерирует STL:
  - габарит модели 100x100 мм;
  - цельное основание (стенд);
  - сглаживание + подавление выбросов + интерполяция сетки для уменьшения "зубчатости".
- Использует локальный кэш DEM (`cache/dem`) для ускорения повторных запросов.

## 2) Архитектура приложения

### 2.1 Модули

- `main.py`
  - точка входа;
  - инициализирует logging и запускает async-бот.
- `bot.py`
  - Telegram-слой (aiogram);
  - команды `/start`, `/help`, `/map`, `/stl`;
  - маршрутизация в режимы PNG/STL;
  - отправка результата пользователю.
- `parse.py`
  - парсинг пользовательского ввода координат;
  - поддержка формата области (2 угла, 4 числа).
- `analyzer.py`
  - вся вычислительная и геоданные-логика:
    - загрузка DEM;
    - fallback между источниками DEM;
    - фильтрация, уклоны;
    - построение PNG;
    - построение STL.

### 2.2 Поток данных (end-to-end)

1. Пользователь отправляет `/map` или `/stl`.
2. Бот запрашивает область в формате:  
   `left_top_lat, left_top_lon; right_bottom_lat, right_bottom_lon`
3. `parse.py` извлекает 4 координаты, проверяет диапазоны.
4. `analyzer.py`:
   - нормализует bbox (north/south/west/east),
   - проверяет максимальный размер области,
   - загружает DEM (кэш -> API -> fallback dem_type).
5. Дальше ветвление:
   - `/map`: фильтрация + уклоны + рендер PNG.
   - `/stl`: геометрическая подготовка + триангуляция + запись STL.
6. `bot.py` отправляет файл пользователю и очищает временный артефакт.

### 2.3 Почему есть fallback источников DEM

Разные источники покрывают разные зоны по-разному (особенно высокие широты).  
Поэтому при отсутствии покрытия в первом источнике выполняется автоматическая попытка других источников.

### 2.4 Почему STL выглядит сглаженным, а не "игольчатым"

В пайплайн STL добавлены шаги стабилизации:
- block-mean downsampling (вместо грубого прореживания);
- заполнение пропусков локальным средним;
- clipping экстремумов по перцентилям;
- gaussian smoothing;
- интерполяционное увеличение сетки (upscale).

Это уменьшает "частокол" в урбанизированных областях и делает модель более пригодной к печати.

## 3) Установка и запуск

### 3.1 Локально

```bash
cd diplom
python3 -m venv venv
source venv/bin/activate   # Windows: venv\\Scripts\\activate
pip install -r requirements.txt
```

Создайте `.env`:

```bash
TELEGRAM_BOT_TOKEN=ваш_токен_бота
OPENTOPOGRAPHY_API_KEY=ваш_api_key
```

Запуск:

```bash
python -m main
```

или

```bash
python main.py
```

### 3.2 Docker

```bash
docker-compose up -d
```

Монтируются директории:
- `./cache` — кэш DEM;
- `./output` — временные результаты.

---

## 4) Форматы команд и ввода

### 4.1 Команды

- `/start` — приветствие и краткая инструкция.
- `/help` — справка по формату.
- `/map` — построить PNG-карту.
- `/stl` — построить STL-модель.

### 4.2 Формат области

Передаются **два угла**:
- левый верхний;
- правый нижний.

Пример:

```text
55.80, 37.50; 55.70, 37.70
```

Допустимо также в виде 4 чисел через пробел.

## 5) API как библиотека

Пример PNG по области:

```python
from analyzer import ReliefAnalyzer

analyzer = ReliefAnalyzer(use_cache=True)
analyzer.build_map_png_by_corners(
    left_top_lat=55.80,
    left_top_lon=37.50,
    right_bottom_lat=55.70,
    right_bottom_lon=37.70,
    output_path="output/map.png",
)
```

Пример STL по области:

```python
from analyzer import ReliefAnalyzer

analyzer = ReliefAnalyzer(use_cache=True)
analyzer.build_relief_stl_by_corners(
    left_top_lat=55.80,
    left_top_lon=37.50,
    right_bottom_lat=55.70,
    right_bottom_lon=37.70,
    output_path="output/relief.stl",
    size_mm=100.0,
    base_thickness_mm=4.0,
    relief_height_mm=16.0,
)
```

## 6) Производительность: реальные замеры

Ниже приведены фактические замеры на текущей машине разработчика для одной и той же географической зоны (район Москвы), но с разным размером области.

### 6.1 Методика

- Python: из `venv` проекта.
- Кэш:
  - **warm-cache**: измерения после прогрева кэша DEM;
  - **cold-cache**: `use_cache=False`.
- Для warm-cache:
  - 4 запуска на каждый тест;
  - первый запуск отбрасывается как прогрев;
  - в таблице: `avg` и `min` по оставшимся 3 запускам.
- Режимы:
  - `map` — PNG;
  - `stl_std` — STL стандарт;
  - `stl_hq` — STL с повышенной детализацией.

### 6.2 Объем данных

Объем данных менялся через размер области:
- `small`: half-size = `0.02°` (bbox `0.04° x 0.04°`)
- `medium`: half-size = `0.06°` (bbox `0.12° x 0.12°`)
- `large`: half-size = `0.10°` (bbox `0.20° x 0.20°`)

Фактические размеры DEM:
- small: `144 x 144`
- medium: `432 x 432`
- large: `720 x 720`

### 6.3 Warm-cache (основной сценарий эксплуатации)

| Case | DEM size | map avg (s) | map min (s) | stl_std avg (s) | stl_std min (s) | stl_std triangles | stl_hq avg (s) | stl_hq min (s) | stl_hq triangles |
|---|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| small | 144x144 | 0.204 | 0.202 | 0.199 | 0.187 | 167,036 | 0.416 | 0.414 | 354,476 |
| medium | 432x432 | 0.331 | 0.331 | 0.266 | 0.255 | 206,076 | 0.441 | 0.438 | 354,476 |
| large | 720x720 | 0.382 | 0.373 | 0.301 | 0.289 | 206,076 | 0.479 | 0.472 | 354,476 |

Вывод:
- PNG режим масштабируется почти линейно по размеру DEM.
- STL-режим в текущих настройках стабилизируется ограничениями финальной сетки (`max_grid_size`, `max_final_grid_size`), поэтому время растет умеренно.
- `stl_hq` примерно в `1.5-2x` тяжелее `stl_std`.

### 6.4 Cold-cache (без кэша, включая запросы к API)

| Case | DEM size | cold map (s) | cold stl_std (s) | DEM source |
|---|---:|---:|---:|---|
| small | 144x144 | 11.731 | 22.085 | SRTMGL1 |
| medium | 432x432 | 11.962 | 23.827 | SRTMGL1 |
| large | 720x720 | 12.448 | 24.457 | SRTMGL1 |

Вывод:
- На cold-cache доминирует сетевой этап (загрузка DEM), а не вычисления.
- Поэтому время зависит от размера области слабее, чем в warm-cache.

## 7) Ограничения и рекомендации

- Не запускайте несколько polling-инстансов одного токена одновременно (получите `TelegramConflictError`).
- Для стабильной работы используйте персональный `OPENTOPOGRAPHY_API_KEY`.
- Слишком большие области отсекаются ограничением bbox (защита от неадекватного объема запроса).
- Для 3D-печати:
  - `stl_std` — быстрее;
  - `stl_hq` — более гладкая геометрия и больше треугольников.
